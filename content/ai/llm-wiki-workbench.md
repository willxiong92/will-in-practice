# 人机协同工作台：用 LLM Wiki 持续编译知识

常见做法是把资料丢进聊天窗口或 RAG，每次提问都从碎片临时拼答案。  
更好的做法是：**你负责收集原料，AI 负责增量维护结构化知识**，下次直接站在已编译结果上回答。

## 这篇内容解决什么问题

- 同一个主题反复问，答案每次都不一样
- 资料越积越多，却无法复用
- 团队只剩聊天记录，没有可交接的方法页

## 三层结构

| 层 | 职责 | 谁维护 |
|---|---|---|
| **Raw** | 原始资料：文章、纪要、截图、数据导出 | 人放入，默认只读 |
| **Wiki** | 编译后的概念页、方法页、对比页、索引 | AI 增量维护，人审核 |
| **Schema** | 规则文件：目录约定、写入流程、质量门槛 | 人维护，AI 遵守 |

一句话：Raw 是事实来源，Wiki 是可复用产物，Schema 是生产规范。

## 最小工作流

```text
采集资料进 Raw
  -> 按 Schema 编译 / 更新 Wiki
  -> 更新 index 与 log
  -> 查询时优先读 Wiki，不够再回 Raw
  -> 形成新判断时写回 Wiki
```

### 采集

- 电脑端：网页剪藏为 Markdown 进 Raw
- 移动端：先用收藏中转，再定期同步到 Raw
- 原则：先保证“进得来”，再优化自动化

### 编译

1. 先读索引，避免重复建页
2. 判断更新旧页还是新建
3. 新旧冲突时保留来源、时间与适用范围，不静默覆盖
4. 补交叉引用
5. 写操作日志，方便回溯

### 查询

- 先 Wiki，后 Raw
- 对客输出前走核验清单
- 新结论要问：要不要写回知识层

## 工具组合怎么选

| 目标 | 更合适的组合 | 说明 |
|---|---|---|
| 本地可控、可双向链接浏览 | 本地库 + Agent 编译器 + 笔记软件 | 适合长期个人/团队母版 |
| 先跑通最小闭环 | 轻量项目空间 + 内容入口 + 任务/资产 | 适合先验证流程，再加深工具 |
| 编码与交付协作 | Coding Agent + IM 桥接 | 适合把执行会话留在协作入口 |

工具会换，**三层分工不要换**。

## 对客时怎么用这套工作台

| 任务 | 工作台动作 | 对客输出 |
|---|---|---|
| 客户诊断 | 从 Wiki 拉诊断框架 + 填入现场证据 | 主结论 + 3 条优先动作 |
| 方案陪跑 | 复用落地方案模板 | 目标、任务、责任、节奏、验收 |
| 复盘 | 用阶段复盘结构汇总 | 变化、卡点、下一步、沉淀项 |

## 完成标准

- 同类问题第二次回答更快，且能引用已有页面
- 新资料进入后有编译记录，不是只堆在聊天里
- 对客材料能追溯到 Wiki 页或 Raw 来源
- 高频任务有 Schema 规则，而不是每次临场发挥

## 风险边界

- 不要把客户隐私、账号密钥直接写入可共享 Wiki
- Schema 写不好时，AI 会“看起来很勤快地制造混乱”
- 本页讲工作台模式，不绑定某一厂商的永久能力清单
