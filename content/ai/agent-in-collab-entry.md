# 把 Coding Agent 接到协作入口

终端里同时开很多 Agent 标签时，常见问题是：会话丢、难检索、难转发、难给反馈。  
对需要持续交付的任务，更稳的做法是：**一个项目一个会话入口，过程留痕，反馈可定位**。

## 适合什么场景

- 需要多轮改代码 / 改文档 / 改方案
- 希望手机端也能跟进任务进度
- 需要把领导/客户反馈一键转给 Agent 继续处理
- 需要把过程分享给同事复盘

不适合：一次性闲聊、或必须完全离线的敏感环境。

## 推荐协作约定

| 约定 | 做法 | 收益 |
|---|---|---|
| 一项目一入口 | 一个群/话题对应一个项目 | 上下文不串 |
| 一任务一会话 | 新任务新开会话，不无限续聊 | 便于归档 |
| 文档优先于长 Markdown 刷屏 | 产出写进可评论文档 | 反馈可定位 |
| 富反馈优于纯文字复述 | 用卡片、表格、批注 | 降低沟通成本 |
| 过程可检索 | 会话留在协作系统 | 方便交接与复盘 |

## 最小落地步骤

1. 选定一个主 Agent（代码或写作均可）
2. 选定一个协作入口（IM / 项目群）
3. 约定：任务描述、完成标准、禁止事项、输出位置
4. 所有关键反馈尽量“原消息转发”，减少二次转述失真
5. 任务结束后沉淀：结论、未决问题、可复用模板

## 对客交付时的额外闸门

- Agent 输出不等于可发送版本
- 涉及客户数据、报价、承诺，必须人工复核
- 不要把内部路径、密钥、未授权截图带进协作群

## 完成标准

- 同事能在不打开你电脑的情况下接上任务上下文
- 关键结论可检索，不只在某次终端滚动历史里
- 反馈能对应到具体段落或动作，而不是“再改改”

## 风险边界

桥接工具与权限模型会变化。公开实践只保留协作模式，具体安装命令与第三方仓库以官方文档为准，并自行评估安全边界。
