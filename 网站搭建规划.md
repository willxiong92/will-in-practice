# Will 的最佳实践网站搭建规划

> 状态：规划稿  
> 日期：2026-07-14  
> 适用范围：AI、FDE、外贸三个知识领域的公开展示与持续沉淀

## 1. 结论

这个网站应定位为“个人最佳实践发布站”，不是把现有本地知识库整体搬到线上，也不是再维护一套知识正文。

推荐采用以下模式：

```text
本地知识库 canonical 母版
        ↓ 内容筛选与脱敏
网站发布稿 content/
        ↓ 自动构建
公开网站
        ↓ 读者反馈与使用数据
回到对应本地知识库迭代
```

网站首期只做三件事：

1. 让访客快速理解 Will 在 AI、FDE、外贸三个领域的实践方向。
2. 让访客按真实任务找到可直接使用的方法、案例和模板。
3. 建立一条不会制造“双份正文”的稳定发布流程。

## 2. 关键假设、风险与可调整项

### 当前假设

- 网站主要服务公开访客、同行、潜在合作方和团队成员，而不是承担内部知识库管理。
- 内容以中文为主，后续根据实际需要增加英文摘要或英文页面。
- 首期不做登录、付费、评论社区和复杂后台。
- 本地知识库继续作为唯一事实来源，网站只保存经过筛选的公开发布稿。

### 主要风险

| 风险 | 影响 | 控制方式 |
|---|---|---|
| 把本地库整体同步上网 | 可能泄露客户信息、个人记录和内部治理内容 | 默认不自动全量同步，只允许白名单内容进入发布区 |
| 三个栏目变成三个文件仓库 | 访客看见很多文章，但不知道该读什么 | 同时提供“领域导航”和“任务导航” |
| 网站正文与本地母版分叉 | 后续维护成本翻倍，内容逐渐冲突 | 每篇发布稿记录 `source_id`、来源项目和复核日期 |
| 一开始做太多功能 | 长时间无法上线验证 | 首期只做内容、导航、搜索、反馈入口和基础数据分析 |
| AI、FDE、外贸边界重叠 | 内容重复，分类越来越混乱 | 领域只作主归属，场景、角色、形式用标签表达 |

### 可调整项

- 网站是偏“个人品牌作品站”还是偏“专业知识工具站”。本规划默认后者占 70%，个人品牌占 30%。
- 托管平台可在 Cloudflare Pages 与 Vercel 之间切换。
- 后续可增加英文版、邮件订阅、AI 问答和会员内容，但不进入首期。

## 3. 网站定位

### 一句话定位

把 AI、FDE 与外贸一线实践，整理成能看懂、能照做、能复用的方法与工具。

### 内容价值层级

网站不应只发布“观点文章”，而要形成四层内容：

| 层级 | 解决的问题 | 典型内容 |
|---|---|---|
| 认知 | 为什么值得关注 | 观点、趋势判断、概念解释 |
| 方法 | 应该怎么做 | 框架、方法论、SOP、检查清单 |
| 实践 | 实际如何落地 | 案例复盘、项目记录、最佳实践 |
| 工具 | 能否直接拿来用 | 模板、提示词、任务卡、工作流 |

优先沉淀“方法、实践、工具”，它们最能体现差异化，也最有复用价值。

## 4. 信息架构

### 顶层导航

```text
首页
├── AI
├── FDE
├── 外贸
├── 实践库
├── 关于 Will
└── 搜索
```

“AI / FDE / 外贸”回答内容属于哪个领域；“实践库”回答读者现在要完成什么任务。两种入口指向同一批内容，不复制正文。

### 三个领域的建议结构

#### AI

- AI 基础认知
- Prompt 与上下文
- Tool、MCP、Agent、Skill
- AI 工作流与自动化
- 团队 AI 推广与治理
- 工具评测与实践复盘

#### FDE

- 新人 onboarding
- 客户诊断
- 方案设计
- 陪跑与落地
- 客户成功 SOP
- 可复用模板与案例复盘

#### 外贸

- 外贸业务基础
- 独立站与获客
- CRM 实施与使用
- 行业研究
- 客户运营
- 外贸 AI 应用

### 实践库的任务入口

- 我想快速学习一个概念
- 我想解决一个工作问题
- 我想照着完成一个任务
- 我需要一份模板或清单
- 我想看真实项目如何落地

### 首页结构

1. 首屏：一句话定位、三个领域入口、主行动按钮“浏览实践”。
2. 本期精选：3 至 5 篇最能代表能力的内容，不按发布时间机械排序。
3. 按任务进入：学习、诊断、执行、复盘、复用。
4. 三个领域地图：展示各领域的知识主线和当前成熟度。
5. 最新实践：只展示最近更新且已公开审核的内容。
6. 关于 Will：说明真实业务背景、实践原则和内容边界。
7. 反馈入口：提交“看不懂、做不了、事实错误、已过时、希望补充”。

## 5. 内容模型

每篇公开内容使用统一元数据：

```yaml
---
title: 使用 AI 做客户诊断的基础流程
slug: ai-customer-diagnosis
domain: ai
topics:
  - customer-success
  - diagnosis
content_type: playbook
audience:
  - csm
  - team-lead
maturity: validated
visibility: public
summary: 用一套可核验的步骤，把零散客户信息转成诊断假设和下一步行动。
source_id: fde/customer-diagnosis-v1
source_project: 团队 FDE 知识库
reviewed_at: 2026-07-14
updated_at: 2026-07-14
---
```

### 必填字段

| 字段 | 用途 |
|---|---|
| `domain` | AI、FDE、外贸的主归属，只能有一个 |
| `topics` | 处理交叉主题，可有多个 |
| `content_type` | `article / guide / playbook / case / template / checklist` |
| `audience` | 告诉谁适合使用 |
| `maturity` | `draft / reviewed / validated / archived` |
| `visibility` | `private / team / public`，只有 `public` 能构建进网站 |
| `source_id` | 关联本地唯一母版，防止出现双份事实来源 |
| `reviewed_at` | 提醒事实复核和过期治理 |

### 每篇内容的页面结构

1. 这篇内容解决什么问题。
2. 适合谁，以及不适合什么场景。
3. 前置条件。
4. 方法或执行步骤。
5. 示例、模板或检查清单。
6. 完成标准。
7. 风险与边界。
8. 更新时间和反馈入口。

## 6. 内容来源与发布流程

### 目录边界

```text
个人网站搭建/
├── site/                  # 网站程序
├── content/               # 经过审核的公开发布稿
│   ├── ai/
│   ├── fde/
│   └── trade/
├── content-registry.yaml  # 来源、状态与发布映射
├── scripts/               # 校验与构建辅助脚本
└── docs/                  # 网站规则、内容规范与决策记录
```

不建议让网站构建程序直接遍历所有本地知识库。正确方式是：从来源项目挑选内容，生成或复制为公开发布稿，完成脱敏与复核后再进入 `content/`。

### 发布门禁

每篇内容上线前必须通过：

- [ ] 有唯一 `source_id` 和来源项目。
- [ ] `visibility: public`。
- [ ] 无客户、员工、合同、账号、密钥、本机路径和内部治理信息。
- [ ] 外部事实有来源或明确标注为经验判断。
- [ ] 有适用对象、使用场景、执行步骤和完成标准。
- [ ] `reviewed_at` 未过期。
- [ ] 页面链接、图片与下载附件有效。

### 后续自动化

当发布动作稳定重复 3 次以上，再增加自动化：

1. 扫描白名单发布稿并校验 frontmatter。
2. 检查敏感关键词、本机路径和失效链接。
3. 生成领域页、标签页、RSS 和 sitemap。
4. 构建预览站，人工确认后合并发布。
5. 将网站反馈整理为待办，回流对应 `source_id`。

## 7. 技术方案

### 推荐方案：Astro + Markdown/MDX + 静态部署

| 组件 | 选择 | 原因 |
|---|---|---|
| 网站框架 | Astro | 面向内容型网站，默认静态生成和较少的浏览器 JavaScript |
| 内容格式 | Markdown 为主，MDX 仅用于确有交互的页面 | 与现有知识库习惯接近，迁移和维护成本低 |
| 样式 | Tailwind CSS + 自有 design tokens | 能快速建设，同时避免被模板样式锁死 |
| 搜索 | 首期使用构建期静态搜索索引 | 无后端、成本低，内容规模扩大后再升级 |
| 部署 | Cloudflare Pages | 可从 Git 自动构建，并为改动提供预览部署 |
| 数据分析 | 隐私友好的轻量分析 | 只关注页面访问、搜索词、入口点击和反馈，不采集多余信息 |

Astro 的官方说明将其定位为内容型网站框架，并默认尽可能在构建期输出页面、减少客户端 JavaScript；Cloudflare Pages 支持 Git 推送后自动构建和预览部署。因此它更贴合当前“内容先行、交互较轻”的网站，而不是为了未来可能出现的功能先上复杂应用架构。

### 为什么暂不首选 Next.js

Next.js 同样支持 MDX 和静态导出，未来若网站需要登录、个性化、复杂表单或服务端 AI 功能，它会更合适。但首期内容站使用它会带来更多应用层概念，收益有限。

### 更简单的替代方案

如果目标只是两周内验证内容是否有人看，可先用现成文档主题搭建最小站点。它上线更快，但个人品牌表现、首页叙事和跨领域任务导航会受模板限制。建议只把它作为临时验证方案，不作为长期主站。

## 8. 视觉与交互方向

**Design Read**：这是一个面向外贸 SaaS、客户成功和 AI 实践者的专业知识发布站，视觉语言应当可信、克制、编辑化，接近现代研究手册，而不是常见的 AI 紫色渐变产品页。

设计参数：

- `DESIGN_VARIANCE: 6`：有清晰编辑感，但不牺牲阅读效率。
- `MOTION_INTENSITY: 3`：只用来表达导航、状态和阅读反馈。
- `VISUAL_DENSITY: 4`：首页有概览密度，正文保持舒适行宽。

建议规则：

- 全站使用浅色主题，冷灰中性色加一种深绿色或钴蓝强调色。
- 首页采用非对称编辑布局，正文页采用稳定的阅读栏与目录。
- 不用 AI 紫色渐变、玻璃卡片堆叠和三张等宽功能卡。
- 三个领域不分别使用三套颜色，避免形成三个割裂的子站；通过图形、编号和文字标签区分。
- 每个领域页应有一张真实的工作场景图、流程图或内容样例，不能只堆文字卡片。
- 动效只用于页面进入、目录定位、筛选反馈和移动端导航，并支持 reduced motion。

## 9. 页面清单与优先级

### MVP 必须有

| 页面 | 作用 |
|---|---|
| 首页 | 讲清定位并把访客导向内容 |
| AI / FDE / 外贸领域页 | 展示知识地图、精选内容和任务入口 |
| 实践库 | 按任务、对象、内容形式筛选 |
| 内容详情页 | 承载方法、案例、模板和完成标准 |
| 关于页 | 建立专业背景与可信度 |
| 搜索页 | 跨三个领域检索 |
| 404 与空结果页 | 提供返回路径和内容建议 |

### 第二阶段再做

- 系列专题与学习路径。
- 邮件订阅与更新摘要。
- 模板下载与版本记录。
- 中英文切换。
- 内容反馈看板。

### 明确暂不做

- 登录与会员系统。
- 在线社区和公开评论。
- 自动抓取所有本地知识库。
- 对公开访客开放的通用 AI 聊天框。
- CRM 行业配置器与 SHOP 行业报告的正式能力入口，除非试点验证通过。

## 10. 分阶段建设计划

### 阶段 0：内容盘点与门禁，2 至 3 天

- 盘点 AI、FDE、外贸现有项目，但不移动或改名。
- 为候选内容标记 `public / team / private`。
- 每个领域选 3 篇首发内容，共 9 篇。
- 建立字段规范、发布检查表和 `content-registry.yaml`。

验收：9 篇内容有唯一来源、明确受众、公开安全结论和成熟度。

### 阶段 1：可用原型，3 至 5 天

- 搭建首页、三个领域页、实践库和正文模板。
- 接入 9 篇真实内容，不使用假文案填充。
- 完成移动端、搜索、404、空结果和基础 SEO。

验收：访客能在 3 次点击内找到一篇适合当前任务的内容。

### 阶段 2：视觉与发布闭环，3 至 5 天

- 完成视觉系统、真实图片与内容图示。
- 接入 Git 预览部署、域名和轻量数据分析。
- 加入内容校验、敏感信息检查和链接检查。
- 用 3 名真实用户完成任务测试。

验收：桌面和手机均无明显布局问题；发布稿可追溯；真实用户能完成查找、阅读和反馈。

### 阶段 3：持续运营

- 每两周发布或更新 1 至 2 篇高价值内容。
- 每月检查搜索词、退出页、反馈和过期内容。
- 每季度重排精选内容和领域知识地图。
- 只有在出现稳定需求后再增加订阅、英文版或 AI 问答。

## 11. 首发内容选择标准

每个领域先选 3 篇，分别覆盖：

1. 一篇代表核心方法论。
2. 一篇代表真实落地实践。
3. 一份可以直接使用的模板或检查清单。

首发不要优先选择“资料最完整”的内容，而应选择：

- 能体现 Will 独特经验。
- 对目标读者有明确任务价值。
- 不依赖大量内部上下文。
- 可以安全公开。
- 能在 5 至 10 分钟内读懂并采取行动。

## 12. 成功指标

首期不以访问量作为唯一目标，重点验证内容是否真正有用：

| 指标 | 判断目的 |
|---|---|
| 领域入口到正文的点击率 | 首页分类是否清楚 |
| 搜索成功率与零结果词 | 内容命名和覆盖是否合理 |
| 模板使用或复制行为 | 内容是否具备实际复用价值 |
| 有效反馈数量 | 是否产生真实工作问题 |
| 过期内容数量 | 内容治理是否可持续 |
| 9 篇首发内容的完成率 | MVP 是否真正形成闭环 |

## 13. 当前建议的下一步

1. 先做“内容资产盘点表”，不要马上写网站代码。网站结构已经足够明确，当前最大不确定性是哪些内容可公开、哪些是唯一母版。
2. 从 AI、FDE、外贸各选 3 篇首发候选，验证内容模型和导航是否成立。
3. 候选内容确认后再搭建 Astro 原型，直接使用真实内容完成首页和正文模板，避免用占位文案做出错误设计判断。

## 15. Codex 与 Grok Build 协作机制

### 角色分工

| 角色 | 负责 | 不负责 |
|---|---|---|
| Will | 业务目标、公开边界、成本与上线决策 | 日常任务拆解和技术验收 |
| Codex 总控 | 需求收敛、架构、内容模型、任务卡、风险判断、diff 审查、测试与最终验收 | 把未收敛的问题直接丢给执行端 |
| Grok Build 执行 | 按已批准任务卡编码、测试、机械修改和聚焦调查 | 自行改变产品范围、内容边界或发布策略 |

### 单任务闭环

```text
Will 提出目标或调整
        ↓
Codex 判断范围、依赖、风险和优先级
        ↓
Codex 编写 Grok Build 任务卡
        ↓
Grok Build 在限定文件范围内执行
        ↓
Codex 检查 diff、运行测试、核对验收标准
        ↓
不通过则生成定向返工卡，通过则更新项目状态
        ↓
涉及公开发布时由 Will 最终确认
```

### 委派规则

适合交给 Grok Build：

- 已确定页面结构后的组件实现。
- 已确定 schema 后的内容导入脚本。
- 样式修复、响应式适配和无障碍修复。
- 单元测试、端到端测试和链接检查。
- 聚焦的故障调查与代码审查。
- 范围明确、可以通过 diff 和测试验收的重复性工作。

保留在 Codex 总控侧：

- 网站定位、产品范围和信息架构。
- 是否公开某类内容的判断。
- canonical 母版与网站发布稿的边界。
- 视觉方向、核心文案和用户路径取舍。
- 安全、隐私、成本和外部发布决策。
- 最终代码接受与阶段验收。

### 任务状态

每个任务只使用以下状态：

```text
candidate → ready → delegated → review → accepted
                                  ↓
                                rework
                                  ↓
                                review
```

- `candidate`：目标存在，但边界或验收尚未明确。
- `ready`：任务卡完整，可以委派。
- `delegated`：Grok Build 正在执行。
- `review`：执行完成，等待 Codex 检查。
- `rework`：未达到验收标准，只返工失败项。
- `accepted`：证据完整，已通过总控验收。

### 审查门禁

Codex 接受 Grok Build 结果前必须确认：

- [ ] 实际修改没有超出 `Allowed Scope`。
- [ ] 没有覆盖或混入 Will 的既有修改。
- [ ] 没有把本地 private、team 内容带入 public 网站。
- [ ] 没有新增未批准的框架、服务、账号或费用。
- [ ] 构建、类型检查、测试和页面检查按风险执行。
- [ ] 桌面端和移动端关键路径均验证。
- [ ] 内容、空状态、错误状态和无障碍不是占位实现。
- [ ] 验收结论由 Codex 基于证据给出，不以执行任务“完成”代替验收。

```text

README.md                 # 项目目标、架构、启动和验收方式
docs/DECISIONS.md         # 已批准的关键决策
docs/ROADMAP.md           # 阶段、任务状态和依赖
docs/CONTENT_POLICY.md    # public/team/private 与脱敏规则
```
