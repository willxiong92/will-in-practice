# 网站类目与内容主线

> 状态：active  
> 更新日期：2026-07-15  
> 依据：飞书「AI知识库 / FDE知识库」结构、本地 `团队FDE-KB`、Wiki `AI智能体` 三层分工、既定公开门禁

## 1. 结论

公开站顶层主类目为：

```text
AI · FDE · 外贸业务 · 国际站 · 独立站
```

- **FDE 保留对外名称**，不改成「客户成功」。客户成功方法是 FDE 域下的一条主线，不是顶栏别名。
- **AI 主线以对客实践为中心**，不是工具说明书合集，也不是纯模型评测。
- **国际站 / 独立站 / 外贸业务** 三分，避免「外贸」一个桶吞掉三条工作流。
- 一篇内容只有一个 `domain`；交叉能力用 `topics`，不复制正文。

## 2. 顶层导航

```text
首页
├── AI              对客场景下的 AI 工作方式与可交付实践
├── FDE             一体化服务与经营增长方法
├── 外贸业务         开发、报价、谈判、履约业务链路
├── 国际站           B2B 平台获客与运营
├── 独立站           自有站增长（SEO / Ads / 转化）
├── 实践库           跨类目按任务筛选
├── 关于
└── 搜索
```

### 边界一览

|---|---|---|---|
| `ai` | AI | 在对客交付中如何可靠地用 AI 完成判断、生产与核验 | 纯内部运维规则、密钥与账号细节、不可脱敏的产品内幕 |
| `fde` | FDE | 如何用经营公式、服务流程和保障体系推动客户结果 | 国际站平台细则（归国际站）、人事绩效、内部编制 |
| `trade_ops` | 外贸业务 | 外贸成交与履约本身怎么跑 | 平台后台操作、独立站技术 SEO |

## 3. AI：对客实践主线

### 3.1 定位修正

| 旧倾向 | 新定位 |
|---|---|
| 模型认知 / 提示词技巧 / 通用工具评测 | **对客交付中的 AI 工作系统** |
| 以「会用哪个模型」为中心 | 以「客户现场要完成什么任务」为中心 |
| 教程清单平铺 | 方案 → 工作台 → 场景 Playbook → 核验 |

飞书 AI知识库当前形态偏工具与接入（Codex、Claude Code、Obsidian、WorkBuddy、账号、接入飞书）。  
公开站保留其资产，但 **重新编成对客实践主线**：访客先看到「能对客交付什么」，再下钻到「用哪套工作台」。

本地 Wiki 已有对齐锚点：

- 三层分工：**对客服务展示层（外部助手）** 与日常协作层分离
- 任务卡矩阵：外贸业务赋能、数字化运营、AI 工具应用等
- `Will AI使用方案`、基础使用一页纸、智能体选型

### 3.2 AI 的 L2 主线（领域页骨架）

| 序号 | L2 主线 | 访客任务 | 飞书 / 本地供给（编译时参考） | 公开注意 |
|---:|---|---|---|---|
| A1 | 对客 AI 使用总方案 | 建立可复制的 AI 工作方式 | Will AI使用方案；AI 基础使用一页纸 | 去掉个人环境与内部路径 |
| A2 | 人机协同工作台 | 搭一套能持续产出的工具链 | Codex / Claude Code / Obsidian / WorkBuddy + IMA / 接入飞书 | 写「组合与职责」，不写密钥与灰产账号攻略 |
| A3 | 对客场景 Playbook | 在真实业务任务里用 AI | 外部助手场景；外贸 AI 工具应用；独立站/询盘/诊断类任务卡 | 优先场景，不优先软件菜单 |
| A4 | 任务分流与 Agent 选型 | 判断何时直聊、何时 Agent、何时人工 | 智能体选型；三层分工公开改写 | 不暴露内部 Agent 配置与私有规则 |
| A5 | 输出核验与风险边界 | 对客材料可核验、可追责 | 核验清单；事实时效判断 | 必须写清不可用场景（隐私、未授权资料） |
| A6 | 环境与接入基线（次级） | 开始前的最小准备 | 国外账号注册；工具安装与接入 | 公开仅保留合规、通用步骤；不写绕过限制的操作 |

### 3.3 AI 域明确降权内容

以下内容 **可以存在于母版**，但默认 **不作为公开首页/精选主推**：

- 纯视频课目录搬运（Codex 全解、Claude Code 教程等）→ 编译为 A2 工作台中的「能力说明 + 完成标准」
- 仅服务个人知识库运维的规则与脚本
- 依赖未授权截图、客户数据或内部产品路线的演示

### 3.4 AI 与其他域的交叉规则

- 「用 AI 做独立站诊断」：主 domain = `indie_site` 或 `ai` 二选一  
  - 若重点是 **诊断步骤与业务结论** → `indie_site`，`topics: [ai]`  
  - 若重点是 **AI 工作流与核验方法** → `ai`，`topics: [indie-site]`
- 「用 AI 辅助 FDE 陪跑」：同理，主归属看「读者带走的是服务方法还是 AI 方法」

## 4. FDE：与飞书知识库对齐的主线

### 4.1 定位

FDE = 一体化客户成功服务体系的公开方法层（经营判断 + 服务动作 + 执行保障 + 复利沉淀）。  
对外仍称 **FDE**；「客户成功方法」是其下 L2，不是顶栏替换名。

> 本地 FDE-KB 声明：团队库首深耕线索侧为 SHOP + AIReach，**不含国际站平台细则**。  
> 因此公开站的 **国际站** 独立成域；FDE 不承接国际站运营手册。

### 4.2 FDE 的 L2 主线（对齐飞书目录）

| 序号 | L2 主线 | 对应飞书 / FDE-KB | 访客任务 | 公开改写要点 |
|---:|---|---|---|---|
| F1 | 服务体系总纲 | `00-服务体系总纲` | 理解生意链路、公式、产品、服务如何串起来 | 泛化产品名；保留链路与判断逻辑 |
| F2 | 经营公式 | `经营公式` 四象限 | 判断增长卡在线索、转化、客单还是毛利 | 公开可用四象限框架；内部指标口径需脱敏 |
| F3 | 产品与能力矩阵 | `产品矩阵` | 客户问题匹配到哪类能力 | 用能力角色表述，避免未公开的内部版本对比原文 |
| F4 | 经营判断与 Skill 调度 | `AWB 经营大脑` | 按经营问题选方法组合 | 公开为「方法群 / 能力群」，不暴露未开放 Agent 配置 |
| F5 | 保障体系 | `保障体系` | 诊断后如何推动客户执行 | 已有候选：执行卡点、陪跑计划 |
| F6 | 客户成功方法 | `客户成功方法` | 阶段复盘、QBR、对客沟通结构 | 去掉内部角色与考核语境 |
| F7 | 平台化能力建设 | `FDE 平台化能力建设` | 如何把单次服务沉淀为可复用资产 | 强调复利机制，不写内部组织政治 |

### 4.3 经营公式四象限（F2 下的固定子槽）

| 象限 | 公开可讲的问题 | 当前母版侧重 |
|---|---|---|
| 更多有效线索 | 线索从哪来、是否有效、如何入库 | SHOP 诊断、主动开发闭环、线索转客户 |
| 更高转化率 | 有线索不成交卡在哪 | 资产阵地、新客转化、沉睡盘活、复购 |
| 更高客单价 | 如何做方案价值与组合 | 骨架优先，等有可公开样稿再填 |
| 更高毛利率 | 如何稳住交付与结构 | 骨架优先，等有可公开样稿再填 |

### 4.4 FDE 高频任务入口（实践库 / 领域页共用）

与 FDE-KB 新人路由对齐，公开改写后可用：

| 任务入口 | 落到主线 |
|---|---|
| 客户没线索 / 线索差 | F2 更多有效线索 |
| 有线索不成交 | F2 更高转化率 |
| 诊断完客户不执行 | F5 保障体系 |
| 要做阶段汇报 / QBR | F6 客户成功方法 |
| 经验要沉淀复用 | F7 平台化 |

## 7. 从飞书 / Wiki 补货顺序

| 优先级 | 域 | 优先主线 | 原因 |
|---:|---|---|---|
| 1 | 独立站 | 增长总图、落地页、SEO 诊断 | Wiki 最厚，公开改写路径清晰 |
| 2 | AI | A1 总方案、A3 对客场景、A5 核验 | 对齐「对客实践」；工具链作支撑 |
| 3 | FDE | F5 保障、F6 复盘、F1 总纲公开版 | 飞书结构完整，但 team 脱敏成本高 |
| 4 | 国际站 | 运营推广、商机交易 | 需统一公开命名与合规复核 |
| 5 | 外贸业务 | 流程、报价谈判 | 事实与版权复核重 |

补货规则：每条 L2 优先凑齐「1 方法 + 1 实践 + 1 模板/清单」，再考虑扩量。

## 8. 非目标

- 不把飞书知识库整库镜像为网站导航  
- 不在公开站顶栏使用「客户成功」替换 FDE  
- 不把 AI 做成纯工具评测站或视频课目录站  
- 不自动同步 `share: team` / private 页  
- 不在公开页暴露本机路径、内部产品未公开代号、客户与人事材料
