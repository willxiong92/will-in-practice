# 公开实践稿写作规范

> 状态：active  
> 更新日期：2026-07-15  
> 适用：从本地 canonical 知识库编译公开文章的作者模式与审核门禁

## 1. 目标

持续生产**来源可靠、公开安全、能解决真实任务**的实践内容。  
不追求篇数；不把网站做成第三方课程或知识图谱的镜像。

## 2. 来源边界

| 类型 | 用途 | 禁止 |
|---|---|---|
| 本地 canonical wiki / FDE-KB 等 | **事实与方法母版** | 直接改写来源库；把 `team`/`private` 当可公开 |
| 八宝周 PPT 知识图谱等材料 | **仅参考**内容结构、知识组织、呈现逻辑 | 照搬原文、标题、案例、图表、独特表述、品牌化模型、金句 |
| 通用管理模型（如常见教练/会议框架） | 须可独立核验；核验后可作为公开常识框架 | 无法确认时自行补全或写成独家发明 |
| 第三方课程/公众号/付费材料 | 至多作线索 | 近义改写规避原创；大段搬运 |

核验规则：

- 易变事实（平台规则、产品功能、价格、版本、合规）→ 官方或第一方来源 + 日期。
- 无法确认 → 写入 `source_coverage.uncertain` 并删除正文相关断言，**不得猜测补全**。
- 八宝周 `structure.md` / transcript **不得**进入 `source_id` / `source_ref` / 公开引用列表。

## 3. 优先五段式（实践类）

每篇**实践导向**文章优先采用：

1. **场景与问题** — 目标读者、使用场景、实际痛点  
2. **核心判断** — 全文最重要结论（先给判断，再展开）  
3. **方法拆解** — 步骤、流程、矩阵或框架  
4. **风险边界** — 常见错误、适用条件、不适用场景  
5. **行动工具** — 清单、模板、Prompt、示例或验收标准  

不要机械套用：

| 内容形态 | 结构建议 |
|---|---|
| 操作指南 / Playbook | 完整五段式 |
| 检查清单 | 场景与问题 + 核心判断（极短）+ 分组清单作行动工具 + 风险边界 |
| 概念解释 | 判断 → 定义边界 → 对照表 → 何时用/不用；工具可弱化但须有完成标准 |
| 实践复盘 | 背景 → 证据 → 判断 → 沉淀动作；禁止虚构数据 |
| 工具模板 | 场景 → 字段说明 → 空白模板 → 验收 |

视觉：网站正文以清晰、易读、移动端优先；**不**把深色知识图谱风套到所有博客页。

## 4. 知识关联（必填块）

每篇在正文末（或紧接行动工具后）提供 **「知识关联」**，且链接必须真实存在：

| 字段 | 要求 |
|---|---|
| 所属主题与内容类型 | 对应 `domain` / L2 货架意图 + `content_type` |
| 前置知识 | 站内已有实践，或写「无强制前置」 |
| 相关方法 | 站内互补方法；禁止为关联而虚构 slug |
| 配套工具 | 本文行动工具摘要，或链到站内模板/清单 |
| AI / FDE / 外贸应用场景 | 按真实适用写 1–3 条；不适用则写「不适用」 |
| 推荐延伸阅读 | 仅站内已发布或本批已登记草稿；草稿互链须标明审核中 |

未发布、不存在、不可公开的页面：**不写链接**，可写「待编译」缺口，不得伪造 URL。

## 5. 内容产品化

每篇至少包含 **一个可直接使用的成果**：

- 检查清单  
- 执行模板  
- Prompt  
- 诊断表  
- 操作示例  
- 验收标准  

来源不足以支撑工具时：

- 在文中或 `source_coverage` 明确标注 **缺口**  
- **不得**凭空生成伪实践、假数据和不可核验案例  

## 6. 状态控制（与现有 schema 对齐）

用户意图中的「草稿默认态」映射到本仓库字段：

| 意图 | 本仓库字段 |
|---|---|
| 未批准草稿 | `maturity: draft` |
| 不公开 | `visibility: private` |
| 待审 | `publication_status: review_required` |

说明：

- schema **没有** `publication_status: draft`，也 **没有** 独立布尔字段 `review_required`。  
- 正式构建只收录 `visibility: public` 且 `publication_status: approved|published`。  
- 完成来源覆盖、原创性、脱敏、链接与事实审核，且独立对抗性审核 ≥ 8 分后，才可建议改为公开候选/批准。  
- 作者会话不得自行宣布批准；审核须独立会话。

推荐 frontmatter 草稿模板：

```yaml
maturity: draft
visibility: private
publication_status: review_required
# reviewed_by: 仅在独立审核通过并批准公开时写入，例如 grok-adversarial-review
```

批准公开时额外写入：

```yaml
maturity: reviewed
visibility: public
publication_status: approved
reviewed_by: grok-adversarial-review
reviewed_at: YYYY-MM-DD
```

## 7. 单篇作者自检（写作结束）

- [ ] 事实来自 canonical，非八宝周正文  
- [ ] 五段式或与内容形态匹配的变体完整  
- [ ] 有可使用的行动工具，或已标注缺口  
- [ ] 知识关联无死链、无虚构站内文  
- [ ] 无客户/员工/合同/账号/本机路径/内部治理  
- [ ] 通用模型已核验或标 `uncertain` 并删除  
- [ ] 仍为 draft / private / review_required  
- [ ] 文末或附属块有 `source_coverage`（审核用）

## 8. 与政策文档的关系

- 可见性与发布状态：`docs/CONTENT_POLICY.md`  
- 类目与货架：`docs/TAXONOMY.md`  
- 工程与启动：`AGENTS.md`  
- 本文件只管 **怎么写公开实践稿**，不替代内容政策。
